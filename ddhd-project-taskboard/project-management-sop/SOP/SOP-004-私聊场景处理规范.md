# SOP-004 私聊场景处理规范

**版本**: v1.0  
**生效日期**: 2026-03-17  
**适用范围**: 所有子Agent  
**强制性**: ✅ 建议性（私聊时灵活处理，但涉及项目时建议遵守）

---

## 1. 目的

规范Agent被私聊时的处理流程，确保私聊场景下也能获取项目上下文、不偏离任务目标。

---

## 2. 私聊场景分类

| 场景类型 | 说明 | 处理方式 |
|----------|------|----------|
| **项目相关私聊** | 用户私聊问项目相关的事 | 先查项目上下文，再回复 |
| **通用咨询私聊** | 技术/策略/创意咨询 | 直接回复，无需查项目 |
| **闲聊私聊** | 日常寒暄、轻松对话 | 自由发挥，保持人设 |
| **任务指派私聊** | 用户私聊派任务 | 按SOP-001执行 |

---

## 3. 标准处理流程

### 3.1 私聊处理检查清单

收到私聊消息时，按以下顺序判断：

```
Step 1: 判断消息类型
├── 是否提及具体项目名称？ → 是 → 项目相关
├── 是否要求产出交付物？ → 是 → 任务指派
├── 是否涉及技术/策略咨询？ → 是 → 通用咨询
└── 都不是 → 闲聊

Step 2: 根据类型处理
├── 项目相关 → 读取项目文档 → 获取上下文 → 回复
├── 任务指派 → 按SOP-001执行
├── 通用咨询 → 直接回复
└── 闲聊 → 保持人设，自由回复

Step 3: 判断是否需要同步
├── 重要决策 → 同步到飞书群
├── 任务进展 → 同步到项目文档
└── 普通交流 → 无需同步
```

### 3.2 项目相关私聊处理

**触发条件**: 私聊中提到项目名称、项目相关需求

**处理流程**:

```
1. 识别项目
   从消息中提取项目名称
   示例："虾虾网站进度如何？" → 项目=虾虾网站

2. 读取项目上下文
   位置: /DDHD_PROJECTS_SOP/PROJECTS/[项目名称]/README.md
   目的: 了解项目当前状态、里程碑、任务

3. 确认用户意图
   用户问的是：
   - 进度查询？
   - 需求变更？
   - 问题反馈？
   - 新任务指派？

4. 针对性回复
   - 进度查询 → 汇报当前状态
   - 需求变更 → 确认变更内容，评估影响
   - 问题反馈 → 记录问题，给出解决方案
   - 新任务指派 → 按SOP-001执行

5. 判断是否需要公开同步
   - 重要变更 → 在群里同步
   - 一般交流 → 无需同步
```

### 3.3 私聊派任务处理

**触发条件**: 用户在私聊中派任务

**处理流程**:

```
1. 确认是否涉及项目
   - 是 → 读取项目文档，按SOP-001执行
   - 否 → 直接执行任务

2. 评估是否需要创建项目文档
   判断标准：
   - 任务复杂度？（复杂→建议创建）
   - 是否需要多Agent协作？（是→建议创建）
   - 是否长期跟进？（是→建议创建）

3. 向用户确认
   "这个任务建议创建项目文档，方便后续跟进。是否创建？"
   或
   "这是一次性任务，我直接处理，无需创建项目。"

4. 执行任务并汇报
```

### 3.4 边界模糊时的处理

当不确定是否需要查项目上下文时：

```
策略: 礼貌询问

回复模板:
"老板，这个问题是针对某个具体项目吗？
如果是，我先去查一下项目文档，确保回复准确。
如果是一般性咨询，我直接回复您。"

用户回复:
- "是针对虾虾网站的" → 查项目文档
- "就是随便问问" → 直接回复
```

## 4. 私聊 vs 群聊的选择

### 4.1 什么情况建议私聊

| 场景 | 原因 |
|------|------|
| 敏感信息 | 不适合公开讨论 |
| 个人事务 | 与其他人无关 |
| 快速确认 | 一句话能说清的事 |
| 非正式讨论 | 想法碰撞阶段 |

### 4.2 什么情况必须群聊

| 场景 | 原因 |
|------|------|
| 项目决策 | 所有人需要知情 |
| 任务交接 | 上下游Agent需要同步 |
| 跨Agent协作 | 需要多人参与 |
| 重要进度汇报 | 总指挥虾需要掌握全局 |

### 4.3 私聊中的重要信息如何处理

```
私聊中产生的重要信息，必须同步到群聊：

示例：
用户私聊说："虾虾网站的需求变更，首页要加一个视频区域"

Agent处理:
1. 私聊中回复："收到，我记录一下变更内容..."
2. 立即在群里同步：
   "📢 需求变更通知：
   项目：虾虾网站
   变更：首页增加视频区域
   影响：需要调整设计稿和开发计划
   @创作虾 @技术虾 请评估影响"
3. 更新项目文档README.md的变更记录
```

## 5. 示例

### 示例1: 项目相关私聊

**用户私聊**: "虾虾网站M1进度怎么样了？"

**Agent处理**:
```
1. 识别项目：虾虾网站
2. 读取：/DDHD_PROJECTS_SOP/PROJECTS/虾虾网站/README.md
3. 获取当前状态：M1.2已完成，M1.3进行中
4. 回复用户：
   "虾虾网站M1进度：
   ✅ M1.1 视觉风格 - 已完成
   ✅ M1.2 角色形象 - 已完成
   🎨 M1.3 首页UI设计 - 进行中（创作虾负责）
   ⏳ M1.4-M1.6 - 待开始
   整体进度：40%"
5. 无需群聊同步（用户只是查询）
```

### 示例2: 私聊派任务（建议创建项目）

**用户私聊**: "帮我做一个完整的品牌升级方案"

**Agent处理**:
```
1. 确认涉及项目：是（品牌升级）
2. 评估复杂度：高，涉及策划、设计、多轮讨论
3. 向用户确认：
   "老板，品牌升级是一个复杂项目，建议创建项目文档，方便：
   - 记录需求和方案
   - 多Agent协作（策划虾+创作虾）
   - 跟踪进度和版本
   是否创建项目文档？"
4. 用户回复："好，创建吧"
5. 创建项目文档，按SOP-001执行
```

### 示例3: 私聊派任务（无需创建项目）

**用户私聊**: "帮我分析一下这个竞品网站"

**Agent处理**:
```
1. 确认涉及项目：否（一次性咨询）
2. 评估复杂度：低，单次分析
3. 直接执行分析
4. 私聊回复结果
5. 无需群聊同步
```

### 示例4: 边界模糊

**用户私聊**: "这个设计稿怎么样？"

**Agent处理**:
```
1. 判断：没有提及项目名称
2. 礼貌询问：
   "老板，这个设计稿是针对虾虾网站的吗？
   如果是，我去看一下项目上下文再评价。
   如果是其他项目，请告诉我项目名称。"
3. 用户回复："是虾虾网站的首页"
4. 读取项目文档，查看设计需求
5. 给出针对性评价
```

### 示例5: 私聊产生的重要决策

**用户私聊**: "虾虾网站不用视频生成了，改用静态图片"

**Agent处理**:
```
1. 识别：这是项目决策变更
2. 私聊回复："收到，这是一个重要变更，我同步到项目组"
3. 立即在群里发送：
   "🚨 重要决策变更：
   项目：虾虾网站
   变更：视频生成功能改为静态图片
   原因：[用户说明的原因]
   影响：
   - 技术虾：取消视频生成技术预研
   - 创作虾：增加静态图片设计任务
   @技术虾 @创作虾 请调整计划"
4. 更新项目文档README.md
5. 更新任务看板
```
