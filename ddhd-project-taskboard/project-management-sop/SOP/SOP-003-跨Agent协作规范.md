# SOP-003 跨Agent协作规范

**版本**: v1.1  
**生效日期**: 2026-03-17  
**适用范围**: 所有Agent  
**强制性**: ✅ 强制（涉及多人协作时必须遵守）

---

## 1. 目的

规范多Agent协作的流程，确保任务交接顺畅、责任清晰、成果可追溯。

---

## 2. 协作场景分类

| 场景 | 说明 | 示例 |
|------|------|------|
| **顺序协作** | A完成后交给B | 策划 → 设计 → 开发 |
| **并行协作** | A和B同时工作 | 前端和后端同时开发 |
| **支援协作** | B协助A完成任务 | 技术虾协助创作虾解决技术问题 |
| **评审协作** | B评审A的工作成果 | 总指挥虾评审设计稿 |

---

## 3. 标准协作流程

### 3.1 顺序协作流程

**【重要更新】交接目录结构说明**

**推荐目录结构（按任务划分，避免混淆）**：
```
公共/跨岗位交付物/
├── M1-001-文案策划-策划交付/     ← 策划完成，交给设计
├── M2-001-视觉设计-设计交付/     ← 设计完成，交给开发
└── M3-001-开发上线-开发交付/     ← 开发完成，交给运营/总指挥
```

**目录命名规范**：
- 格式：`[里程碑ID]-[任务名]-[岗位]交付/`
- 示例：`M1-003-首页设计-设计交付/`
- 禁止：一个任务目录下混合多个岗位的交付物

---

#### 协作流程

```
阶段1: 任务启动
├── 总指挥虾在群里@下游Agent
├── 说明"上游Agent完成XX后会交给你"
└── 下游Agent提前了解项目上下文

阶段2: 上游Agent执行
├── 在自己工作目录执行（如：策划/策划方案/）
├── 完成后按SOP-002同步信息
└── 在项目文档中标注"已完成，待交接"

阶段3: 任务交接（关键步骤）
├── 上游Agent将交付物复制到跨岗位交付物目录
│   例如：策划完成方案后
│   cp 策划/策划方案/首页策划.md 公共/跨岗位交付物/M1-003-首页设计-策划交付/
├── 上游Agent在群里@下游Agent
│   "@创作虾 策划已完成，文件在公共/跨岗位交付物/M1-003-首页设计-策划交付/"
├── 下游Agent从跨岗位交付物目录读取
└── 总指挥虾更新任务看板

阶段4: 下游Agent执行
├── 读取上游交付物（从公共/跨岗位交付物/[任务]/[岗位]交付/）
├── 在自己工作目录执行（如：设计/设计稿/）
├── 完成后进入下一轮交接或汇报
└── 交付物复制到下一环节的跨岗位交付物目录（如：公共/跨岗位交付物/M1-003-首页设计/设计交付/）
```

**【新增】文件创建确认机制**

交接时必须确认：
```
□ 1. 文件已保存到自己的工作目录（不是默认目录）
□ 2. 文件已复制到跨岗位交付物/[任务]/[岗位]交付/
□ 3. 文件名清晰可识别（包含版本号如-v1）
□ 4. 已在群里@下游Agent并说明文件位置
□ 5. 下游Agent已回复"收到"
```

### 3.2 并行协作流程

```
阶段1: 任务拆分
├── 总指挥虾明确各Agent职责边界
├── 确定协作接口（谁提供什么、给谁）
└── 建立协作群/频道

阶段2: 并行执行
├── 各Agent独立工作
├── 每日同步进度（飞书群）
└── 遇到问题立即@相关人

阶段3: 成果整合
├── 各Agent提交成果
├── 总指挥虾或指定Agent整合
└── 验证协作接口是否匹配

阶段4: 整体交付
├── 整合后的成果提交
├── 总指挥虾验收
└── 同步到项目文档
```

### 3.3 支援协作流程

```
触发: A遇到技术难题，需要B协助

流程:
1. A在群里@B，说明问题
   "技术虾，我在设计交互时遇到XX问题，能否支援？"
   当前工作目录：创作虾/设计稿/首页.fig

2. B确认可以支援，了解上下文
   - 读取相关项目文档（README.md, 决策日志.md）
   - 读取A的工作目录文件（如有权限）
   - 与A私聊或群里沟通细节

3. B协助解决问题
   - 在自己的工作目录创建支援文档（如：开发/支援记录/）
   - 或在公共/讨论归档/中记录解决方案
   - 完成后在群里同步结果

4. A继续完成任务
   - 使用B提供的方案
   - 在成果中致谢B的协助

注意: 
- 支援协作不改变任务归属，A仍是任务负责人
- 支援文档统一放在公共/讨论归档/或各自工作目录的支援记录/
```

### 3.4 评审协作流程

```
触发: 成果完成，需要评审

流程:
1. 提交人发送评审请求
   "@总指挥虾 设计稿已完成，请评审"
   附带: 成果文件 + 设计说明

2. 评审人进行评审
   - 检查是否符合需求
   - 检查是否符合规范
   - 记录问题和建议

3. 评审反馈
   - 通过: "✅ 评审通过，可以进入下一阶段"
   - 需修改: "⚠️ 评审意见：[具体问题]"

4. 修改与复评（如需）
   - 提交人根据意见修改
   - 再次提交评审

5. 评审完成
   - 更新项目文档状态
   - 进入下一阶段
```

## 4. 协作 etiquette（礼仪）

### 4.1 @人的规范

| 场景 | 如何@ | 示例 |
|------|-------|------|
| 需要对方立即响应 | @用户名 + 说明紧急程度 | "@创作虾 紧急：设计稿今晚需要" |
| 只需对方知晓 | @用户名 | "@技术虾 设计方案已确定，请知悉" |
| 需要多人协作 | @多人 | "@创作虾 @技术虾 请两位协作完成XX" |

### 4.2 交接的规范

```
交接时必须包含:
□ 交付物文件/链接
□ 相关文档位置
□ 已完成的成果说明
□ 遗留问题/注意事项
□ 下一步建议
```

### 4.3 支援的规范

```
请求支援时:
□ 说明问题背景
□ 说明已尝试的解决方案
□ 明确需要什么帮助
□ 期望的完成时间

提供支援时:
□ 确认理解问题
□ 提供清晰的解决方案
□ 说明方案的局限性（如果有）
□ 后续有问题可以继续问
```

## 5. 禁止行为

| 禁止行为 | 后果 | 正确做法 |
|----------|------|----------|
| 私聊交接重要任务 | 总指挥不知道进度 | 在群里@交接 |
| 交接不说清楚上下文 | 下游Agent一头雾水 | 按交接清单说明 |
| 被交接后不确认 | 上游以为已接收 | 回复"收到，我来看一下" |
| 协作中私自改需求 | 返工、进度延误 | 改需求必须经过总指挥虾 |
| 支援后抢任务归属 | 责任不清 | 明确支援不改变任务归属 |

## 附录A: 子Agent调度前置检查清单【新增 - 强制要求】

**适用范围**: 总指挥虾调度所有子Agent时
**强制性**: ✅ 强制（子Agent必须完成检查后才能执行任务）
**模板位置**: `TEMPLATES/agent-dispatch-checklist.md`

### A.1 前置检查清单（5项强制检查）

子Agent在开始执行任务前，必须完成以下检查，完成后回复"✅ 前置检查完成"：

| 序号 | 检查项 | 目的 | 确认标准 |
|------|--------|------|----------|
| 1 | 读取 MEMORY.md | 了解角色定位、长期记忆、核心规范 | 能说出自己的角色和主要职责 |
| 2 | 读取近期记忆文件 | 了解最近3天的上下文，避免信息断层 | 知道最近处理过什么任务 |
| 3 | 检查可用技能 | 确认有哪些技能工具可用 | 能列出2-3个可用技能名称 |
| 4 | 读取项目文档 | 了解项目背景、目标、约束条件 | 能说出项目目标和当前阶段 |
| 5 | 确认接收的交付物 | 了解输入材料的完整性和质量 | 已查看并理解上游交付物内容 |

### A.2 调度指令格式

**总指挥虾必须使用以下格式调度子Agent**：

```
@[Agent名] 新任务派发

⚠️ 前置检查清单（必须完成，完成后回复"✅ 前置检查完成"）：
□ 1. 读取 [MEMORY.md路径]
□ 2. 读取 [memory/路径] 下最近3天的记忆文件（按文件名日期排序）
□ 3. 检查 [skills/路径] 目录下可用技能
□ 4. 读取 [项目README.md路径]（如有项目关联）
□ 5. 查看前置交付物：[跨岗位交付物路径]（如有前置任务）

未完成以上检查，禁止开始执行任务。

---

🎯 任务执行指令
【任务背景】...
【具体要求】...
【交付位置】...
【发送规则】...
```

### A.3 为什么强制要求前置检查？

| 问题现象 | 根因 | 前置检查解决方式 |
|----------|------|------------------|
| 子Agent输出质量不稳定 | 缺乏长期记忆和上下文 | 检查项1+2：读取MEMORY.md和近期记忆 |
| 重复造轮子、不知道用技能 | 不了解可用工具 | 检查项3：检查skills目录 |
| 偏离项目目标 | 不了解项目背景 | 检查项4：读取项目README |
| 看不懂上游交付物 | 没查看输入材料 | 检查项5：确认接收的交付物 |

### A.4 禁止行为

| 禁止行为 | 后果 | 正确做法 |
|----------|------|----------|
| 不发送前置检查清单 | 子Agent遗漏关键上下文 | 必须发送清单，等待确认 |
| 子Agent未确认就开始执行 | 输出质量无法保证 | 必须等待"✅ 前置检查完成"回复 |
| 清单项目模糊（如不写具体路径） | 子Agent找不到文件 | 必须写完整绝对路径 |
| 总指挥虾不验收确认结果 | 无法验证检查是否真实完成 | 抽查确认子Agent是否真读了文件 |

### 3.5 视觉类交付物特殊规范【新增】

**适用Agent**: 创作虾（👩🏻‍🎨 负责图片、视频生成）

#### 强制要求

创作虾产出**图片、视频类文件**时，必须：

```
□ 1. 将文件保存到正确位置（设计/设计稿/ 或 设计/素材/）
□ 2. 将文件复制到跨岗位交付物/[任务]-[岗位]交付/
□ 3. 【强制】使用 message 工具发送文件到群聊
   - 目的：便于总指挥虾和用户实时审阅
   - 格式：发送文件 + 简要说明
□ 4. 在群里@总指挥虾或用户，说明设计要点
□ 5. 等待审阅反馈后再进入下一步
```

#### 示例流程

**创作虾完成设计后**：

```
1. 保存文件到项目目录
   → PROJECTS/虾虾网站/设计/设计稿/首页-v1.png

2. 复制到交接区
   → PROJECTS/虾虾网站/公共/跨岗位交付物/M1-003-首页设计-设计交付/

3. 【关键步骤】使用 message 工具发送文件
   {
     "action": "send",
     "channel": "feishu",
     "filePath": "media/projects/虾虾网站/设计/设计稿/首页-v1.png"
   }

4. 在群里发送文字说明
   "@总指挥虾 首页UI设计完成！
   🎨 设计要点：
   - 采用科技蓝+珊瑚橙配色
   - 卡片式布局，圆角16px
   - 响应式设计，支持多端
   请审阅，如有修改意见告诉我~"

5. 等待审阅反馈
   - 通过：进入下一步交接给技术虾
   - 需修改：根据反馈调整
```

#### 为什么强制发送？

| 原因 | 说明 |
|------|------|
| **实时审阅** | 图片视频需要视觉确认，不能只看文件路径 |
| **及时反馈** | 发现问题可立即修改，避免下游返工 |
| **决策依据** | 总指挥虾和用户需要看到实际效果才能决策 |
| **防止遗漏** | 避免文件存了但没人知道的情况 |

#### 禁止行为

| 禁止 | 后果 | 正确做法 |
|------|------|----------|
| 只存文件不发送 | 没人知道完成了，无法审阅 | 必须发送文件到群聊 |
| 只发文字说明 | 看不到实际效果 | 必须附带图片/视频文件 |
| 私聊发送 | 群成员无法同步知晓 | 必须在项目群发送 |
